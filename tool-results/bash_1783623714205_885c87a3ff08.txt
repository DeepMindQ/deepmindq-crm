14. Index and Constraint Strategy

-   Unique or near-unique indexes: ingestion_batches.file_hash, import_profiles.header_hash, companies.domain when present, inbound_messages.provider_message_id, outbound_messages.provider_message_id, users.email_address.

-   Search indexes: contacts.email_address, contacts.linkedin_url_normalized, contacts.name_normalized, companies.name_normalized, companies.domain, companies.linkedin_url_normalized, contacts.status, contacts.email_health_status, custom_field_definitions.internal_key, custom_field_values.normalized_value_text.

-   Foreign key indexes: all parent-child relationships used for search and joins.

-   JSONB indexes: contacts.custom_fields, companies.custom_fields, and settings payloads only where needed, with GIN indexing if queryable.

-   Temporal indexes: audit_logs.created_at, email_health_checks.checked_at, outbound_messages.scheduled_for, inbound_messages.received_at, ai_generation_runs.started_at.

-   Partitioning is optional at first but can later be added for audit_logs, outbound_messages, inbound_messages, email_health_checks, and ai_generation_runs if volume grows sharply.

15. Recommended Constraints

-   A contact must always point to a company, even if the company is a placeholder.

-   A company must always have a normalized name for duplicate detection.

-   A batch file hash should never be imported twice without an explicit override path.

-   A suppression entry should block drafting and sending until removed by an authorized user.

-   A revoked consent state or a hard suppression should block all future sends immediately.

-   A health score below the block threshold should prevent queueing unless an authorized override is recorded.

-   A draft should not move to outbound_messages until it has been approved.

-   Approved drafts should create outbound_messages in queued state instead of bypassing the queue.

-   An outbound message should not send outside the configured working window unless the user explicitly overrides the queue.

-   Dry-run mode should record outbound_messages but skip actual Graph send calls.

17. Practical Notes

-   Keep the schema wide enough for current spreadsheet reality, but not so wide that every field needs a migration.

-   Favor JSONB for source-specific custom data and relational columns for everything that must be filtered, sorted, or reported often.

-   Use custom_field_definitions plus custom_field_values when a custom column needs validation, filtering, or prompt access.

-   Keep event tables append-only wherever possible.

-   Use audit logs for who changed what and when, but use history tables for operational traces that need fast lookup.

-   The schema should support the web app without requiring a separate mobile data model.
